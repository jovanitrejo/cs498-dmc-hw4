Q1: FoundationDB separates transaction management from storage so each part can specialize in a different job and scale independently. The TMS handles coordinating transactions, such as conflict checking, committing, and guaranteeing ACID. The Storage System is responsible for actually storing key-value data and serving reads. This separation makes the system modular, making each component much easier to scale when needed, which makes maintaining correctness easier.
Q2: The resolver detects read-write conflicts between concurrent transactions. Each transaction is given a commit version, and the resolver checks the transaction's read conflict ranges against the write conflict ranges of transactions that committed at newer version since that transaction's read version. If there's an overlap, the transaction is rejected and the commit is not fully made.
Q3: The read version tells a transaction which snapshot of the DB it should read from. All reads in that transaction are interpreted as if they happened at one logical point in time, even if the system is distributed and other transactions are committing concurrently. This gives strong consistency for transactional reads, because the transaction sees a stable, consistent snapshot rather than a mix of stale and new data.
Q4: Snapshot reads return data from the same consistent snapshot as normal reads, but they do not add read conflict ranges, meaning they won't cause the transaction to conflict if another transaction later writes to that data. However, standard reads track these ranges and will trigger a conflict if the data is modified before the commit. Snapshot reads would be used when read-data is needed for non-essential or advisory purposes such as logging, data analytics, and IoT data ingestion. Standard reads should be used the read value must be protected from concurrent writes to ensure correctness.
Q5: FDB's recovery is cheap because the LogServers durably store all committed mutations in a replicated write ahead log (WAL), while StorageServers asynchronously pull and apply the updates in the background during normal operation. This means the system always has an up to day log of committed transactions independent of the storage state, so recovery doesn't depend on replaying a traditional redo log from scratch. When a failure occurs, a new Transaction System can quickly determine the latest version from Storage Servers and then use LogServers to catch up only the missing updates.
